Section 01

Data Pipeline & Universe Construction

The data source is the NBA Stats API via the nba_api Python library. I pulled all available game records using LeagueGameFinder(league_id_nullable='00') and filtered to two complete seasons: 2023–24 and 2024–25. A critical design choice: the data is sorted chronologically before any feature computation. Rolling statistics are computed with a one-game lag (via .shift(1)) so that a team's current-game performance never leaks into its own features.

Data Flow — From Raw API to Feature Matrix LeagueGameFinder → sort by GAME_DATE → compute rolling(window).mean() on shift(1)
→ drop NaN rows (first {window} games per team have no history) → 5,109 usable rows
→ pivot HOME / AWAY → merge on GAME_ID → 2,536 matched pair rows
→ add differential features → chronological 80/20 split → Train: 2,028 | Test: 508

Star Player Data Augmentation

Using LeagueDashPlayerStats(season='2024-25'), I identified the top 2 scorers by PPG per team. For each player, I pulled their full PlayerGameLog and computed a lagged 15-game rolling average of their PLUS_MINUS. The decision to track top 2 rather than top 1 was deliberate: a secondary star injury moves the signal in a way that pure top-scorer tracking misses.

Section 02

Feature Engineering & Window Selection

Rather than picking 10 games arbitrarily, I ran a chronological cross-validation sweep over the candidate set {5, 7, 10, 15, 20}. Window w=15 was selected at 60.67% holdout accuracy.

Fig. 3 — Rolling Window Cross-Validation Accuracy

The key design principle is that differential features are more predictive than raw values for logistic regression. Instead of asking the model to infer the gap between two teams from separate HOME and AWAY coefficients, we compute it explicitly:

Differential Feature Construction PM_DIFF = HOME_PM_ROLL − AWAY_PM_ROLL
WIN_RATE_DIFF = HOME_WIN_RATE_ROLL − AWAY_WIN_RATE_ROLL
STAR_PM_DIFF = HOME_STAR_PM_ROLL − AWAY_STAR_PM_ROLL
REST_DIFF = HOME_REST_DAYS − AWAY_REST_DAYS
μ(PM_DIFF) = −0.16 | σ = 9.53 | range: [−33.1, +29.9]

Section 03

Model Architecture & Coefficient Analysis

The primary model is a logistic regression with L2 regularisation (solver='liblinear'), trained on 14 features using a strictly chronological 80/20 split. The model outputs a probability P(HOME_WIN = 1 | X).

Logistic Regression Decision Boundary P(Y=1 | X) = σ(β₀ + β₁·HOME_PM_ROLL + β₂·AWAY_PM_ROLL + ... + β₁₄·IS_PLAYOFFS)
where σ(z) = 1 / (1 + e−z)

Intercept: absorbs base home-court advantage.
Classification threshold: P > 0.50 → predict HOME WIN

The Baseline Problem: The baseline home-win rate in the training set is 68.22%. The logistic regression achieves 66.54% on the test set. The model is not simply "worse"—it is producing calibrated probabilities, correctly identifying away wins with high confidence, which a naive "always pick home" rule cannot do. Accuracy is the wrong metric for expected betting edge.
Section 04

Backtesting Framework & Synthetic Odds Engine

A proper backtest requires a market against which to find edge. I constructed a synthetic odds proxy using the WIN_RATE_ROLL differential—deliberately choosing a signal different from PM_ROLL, which is the model's primary predictor.

Synthetic Moneyline Construction & Edge Logic estimated_spread = (WIN_RATE_DIFF × 30) + 2.5
fair_prob_home = Φ(estimated_spread / 13.5) ← normal CDF with σ ≈ 13.5 pts
market_prob_home = fair_prob_home × 1.025 ← add 2.5% bookmaker vig

if (my_prob_home − market_prob_home) > MIN_EDGE (= 0.05):
  bet HOME at home_payout (stake = $50)

Section 06

Playoff Regime Analysis: Why the Regular Season Model Fails

6.1 — Structural Differences: Regular Season vs. Playoffs

The NBA playoffs are not simply an extension of the regular season. They represent a fundamentally different information environment. Understanding these differences is prerequisite to building any model that doesn't silently degrade when applied to playoff games.

Regular Season Characteristics
Schedule density~4 games/week possible
Rest patternsHighly variable
Game planningGeneralised
Star minutesManaged (Load rest)
Home win rate~56–58%
PaceHigher (~100+ poss)
Playoff Characteristics
Schedule density1–2 games/week typical
Rest patternsConsistent — series rhythm
Game planningDeep scouting vs 1 opp
Star minutesMaximised
Home win rate~63–65% (amplified)
PaceSlower (~96–98 poss)

The most important structural change is the series context. In the regular season, each game is independent. In the playoffs, Game 4 of a series is conditioned on who won Games 1–3, what adjustments coaches made, and whether star players have been worn down by specific defensive assignments. None of this series-state information is captured in the current feature set.

6.2 — What the Current Model Gets Right in Playoffs

The IS_PLAYOFFS flag (coefficient +0.1498) allows the model to shift its effective intercept upward when predicting playoff games — increasing the base probability assigned to the home team. This captures the broad "playoffs favour home court" effect.

Star player availability (STARS_PLAYING) becomes more informative in the playoffs. In the regular season, a star may be rested on short notice without strategic intent. In the playoffs, a star's absence is almost always due to injury — a genuine and persistent disadvantage.

Series state is the single highest-value addition. A team leading 3–0 in a series will have: higher defensive intensity (trying to close), potential star minute management (holding back for later rounds if leading), and opponent desperation effects. A model blind to 3–0 vs. 0–3 is missing the primary driver of expected outcome in late-series games.

6.4 — Playoff Betting Performance Analysis

The 95 playoff bets placed represent a distinct regime from the 189 regular-season bets. The IS_PLAYOFFS flag being active means the model applies its full 14-feature vector with the playoff intercept shift, identifying edges primarily in games where team-strength differentials are large enough to overcome the amplified home-court prior. Playoff sample size caveat: 95 bets across one playoff period is an insufficient sample to draw statistically significant conclusions about playoff-specific alpha.
