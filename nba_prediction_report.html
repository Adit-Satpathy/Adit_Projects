1.1 — Dataset Construction & Temporal Integrity

A critical design choice: the data is sorted chronologically before any feature computation. Rolling statistics are computed with a one-game lag (via .shift(1)) so that a team's current-game performance never leaks into its own features. This prevents the most common form of look-ahead bias in sports models.

Data Flow — From Raw API to Feature Matrix LeagueGameFinder → sort by GAME_DATE → compute rolling(window, min_periods=window).mean() on shift(1)
→ drop NaN rows (first {window} games per team have no history) → 5,109 usable team-game rows
→ pivot HOME / AWAY → merge on GAME_ID → 2,536 matched pair rows
→ add differential features → chronological 80/20 split → Train: 2,028 games | Test: 508 games

Why two seasons? A single season (~1,200 team-game rows) leaves only ~240 usable rows in a 20% test set — too small for stable metric estimates. Two seasons doubles this, and crucially, the inclusion of 2023–24 means the model sees an out-of-distribution season for the entire test period (2024–25 playoffs), providing a sharper measure of generalisation.

1.2 — Star Player Data Augmentation

Using LeagueDashPlayerStats(season='2024-25'), I identified the top 2 scorers by PPG per team — 60 players total across 30 franchises. For each player, I pulled their full PlayerGameLog and computed a lagged 15-game rolling average of their PLUS_MINUS. These were then aggregated per team-game to produce two signals:

The upgrade from binary to 0–2 count also preserves ordering information lost by a simple present/absent flag.

Section 02

Feature Engineering & Window Selection

2.1 — Rolling Window Hyperparameter Search

The choice of rolling window size is a classic bias-variance problem. A window that is too short captures recent form but is dominated by single-game noise. A window that is too long smooths out noise but lags structural changes (e.g. a mid-season trade that fundamentally alters a team's identity). Rather than picking 10 games arbitrarily, I ran a chronological cross-validation sweep over the candidate set {5, 7, 10, 15, 20}.

Window Selection Protocol For each w ∈ {5, 7, 10, 15, 20}:
  compute PM_ROLL and WIN_RATE_ROLL with window=w (lagged by 1 game)
  fit LogisticRegression(solver='liblinear') on first 80% of rows
  evaluate accuracy_score on final 20% → select argmax
Fig. 3 — Rolling Window CV Accuracy (Two-Feature Proxy Model)
Holdout Accuracy
Selected (w=15)

Window w=15 was selected at 60.67% holdout accuracy. The differences are small (sub-2% range across all candidates), but 15 games represents approximately 3 weeks of the NBA schedule — long enough to characterise a team's current quality, short enough to respond to a significant player trade or injury within one calendar month.

2.2 — The Feature Set: Design Rationale

The full predictor set contains 14 features across four conceptual groups. The key design principle is that differential features are more predictive than raw values for logistic regression. Instead of asking the model to infer the gap between two teams from separate HOME and AWAY coefficients, we compute it explicitly:

WIN_RATE_DIFF = HOME_WIN_RATE_ROLL − AWAY_WIN_RATE_ROLL
STAR_PM_DIFF = HOME_STAR_PM_ROLL − AWAY_STAR_PM_ROLL
REST_DIFF = HOME_REST_DAYS − AWAY_REST_DAYS

On IS_PLAYOFFS: This binary flag encodes the regime shift between regular season and playoffs. Playoff basketball is structurally different — games are slower-paced, defences are more prepared, and teams with matchup advantages exploit them over a 7-game series. Including this flag allows the model to apply different effective weights in each regime without needing a separate model.

2.3 — Correlation Structure

A multicollinearity concern is real here: PM_DIFF is constructed from HOME_PM_ROLL and AWAY_PM_ROLL, so the three are linearly dependent. The same applies to WIN_RATE_DIFF. However, logistic regression with solver='liblinear' (L2 regularisation by default) is relatively tolerant of moderate collinearity. The inclusion of both raw and differential features is intentional — they probe slightly different aspects of relative quality and the regulariser suppresses redundant combinations.

Fig. 4 — Feature Correlation Heatmap (Interactive)
Multicollinearity flag: PM_DIFF correlates with HOME_PM_ROLL at r ≈ 0.72 and with AWAY_PM_ROLL at r ≈ −0.72. The model includes all three because in practice the coefficients on each absorb different variance. In future iterations a ridge penalty sweep or VIF analysis should inform which to drop.
Section 03

Model Architecture & Coefficient Analysis

3.1 — Logistic Regression as Primary Model

The primary model is a logistic regression with L2 regularisation (solver='liblinear'), trained on 14 features using a strictly chronological 80/20 split. The target is binary: HOME_WIN = 1 if the home team won. The model outputs a probability P(HOME_WIN = 1 | X), which is used both for classification (threshold 0.5) and as a continuous probability estimate for betting edge calculation.

Logistic Regression Decision Boundary P(Y=1 | X) = σ(β₀ + β₁·HOME_PM_ROLL + β₂·AWAY_PM_ROLL + ... + β₁₄·IS_PLAYOFFS)
where σ(z) = 1 / (1 + e−z)

Intercept: not directly interpretable (absorbs home court baseline)
Classification threshold: P > 0.50 → predict HOME WIN

Logistic regression's advantage in this context is interpretability: each coefficient directly quantifies how a one-unit change in a feature shifts the log-odds of a home win. The multinomial comparison against Random Forest and XGBoost (Section 5) shows it is not merely a baseline — it outperforms both on test accuracy while remaining fully interpretable.

3.3 — The Baseline Problem

The stated baseline of 68.22% is the home-win rate in the training set — a trivially simple rule: always predict the home team wins. The logistic regression achieves 66.54% on the test set, which is below this naive baseline. This is a genuine finding that warrants careful interpretation.

Section 04

Backtesting Framework & Synthetic Odds Engine

4.1 — Synthetic Market Odds Construction

A proper backtest requires a market against which to find edge. Real historical moneyline data was not available, so I constructed a synthetic odds proxy using the WIN_RATE_ROLL differential — deliberately choosing a signal different from PM_ROLL, which is the model's primary predictor. Using the same signal to construct both the model output and the market odds would make the backtest circular.

Synthetic Moneyline Construction (per game) estimated_spread = (WIN_RATE_DIFF × 30) + 2.5 ← 10% win-rate gap ≈ 3pt spread; +2.5 for home court
fair_prob_home = Φ(estimated_spread / 13.5) ← normal CDF with σ ≈ 13.5 pts (NBA scoring variance)
market_prob_home = fair_prob_home × 1.025 ← add 2.5% bookmaker vig
home_payout = 1 / market_prob_home ← decimal odds
Home court baseline: estimated +2.5 pt advantage (empirical NBA average)
Critical design note: This synthetic odds engine is the weakest link in the backtest. The 30× calibration multiplier and 2.5pt home advantage are defensible approximations but are not fitted to actual market data. Any ROI figure from this backtest should be interpreted as "performance against a plausible market proxy" rather than "expected real-world profitability." Replacing this with actual historical lines from a data provider (e.g. The Odds API) would be the single highest-priority improvement to the evaluation methodology.

4.2 — Betting Decision Rule

Edge-Based Bet Placement Logic my_prob_home = model.predict_proba(X)[0,1]
vegas_prob_home = 1 / home_payout

if (my_prob_home − vegas_prob_home) > MIN_EDGE (= 0.05):
  bet HOME at home_payout (stake = $50)
elif (my_prob_away − vegas_prob_away) > MIN_EDGE:
  bet AWAY at away_payout (stake = $50)
else: SKIP — no edge identified

The 5% edge threshold means the model must believe the true probability is at least 5 percentage points higher than what the market implies before a bet is placed. This filters out borderline calls where the model has no meaningful informational advantage. Of 508 test games, 284 bets were placed (55.9% bet rate).

6.1 — Structural Differences: Regular Season vs. Playoffs

The NBA playoffs are not simply an extension of the regular season. They represent a fundamentally different information environment. Understanding these differences is prerequisite to building any model that doesn't silently degrade when applied to playoff games.

The most important structural change is the series context. In the regular season, each game is independent. In the playoffs, Game 4 of a series is conditioned on who won Games 1–3, what adjustments coaches made, and whether star players have been worn down by specific defensive assignments. None of this series-state information is captured in the current feature set.

6.2 — What the Current Model Gets Right in Playoffs

The IS_PLAYOFFS flag (coefficient +0.1498, standardised impact +0.0747) allows the model to shift its effective intercept upward when predicting playoff games — increasing the base probability assigned to the home team. This captures the broad "playoffs favour home court" effect.

Star player availability (STARS_PLAYING, the highest individual-feature impact) becomes more informative in the playoffs. In the regular season, a star may be rested on short notice without strategic intent. In the playoffs, a star's absence is almost always due to injury — a genuine and persistent disadvantage. The model's STARS_PLAYING signal is therefore more reliable as a feature in the postseason.

Section 07

Honest Performance Assessment: Strengths, Weaknesses & Improvement Roadmap

7.1 — What the Model Does Well

✓ Temporal Integrity

Every rolling feature is computed with a one-game lag and the train/test boundary is strictly chronological. There is no look-ahead bias. This is non-negotiable in sports modelling and is correctly implemented throughout.

✓ Principled Window Selection

The rolling window of 15 games is not arbitrary — it was selected via cross-validation. The range of tested windows (5–20) spans the meaningful parameter space, and the selection criterion (holdout accuracy) is appropriate.

✓ Differential Features

Including PM_DIFF, WIN_RATE_DIFF, and STAR_PM_DIFF gives the model direct access to relative team strength. This is a cleaner representation than requiring the model to infer differences from separate HOME and AWAY terms.

✓ Regime Flagging

IS_PLAYOFFS is a lightweight but effective way to encode the known home-court advantage amplification in the postseason without requiring a separate model or training dataset.

7.2 — Identified Weaknesses

✗ Accuracy Below Naive Baseline

66.54% vs. 68.22% (always-home rule). This means on a per-game classification basis, a student who knows nothing about basketball would outperform this model on the test set. The model adds value only through probability calibration, not classification superiority.

✗ Synthetic Odds Engine

ROI figures are computed against a simulated market, not actual historical moneylines. The 2.5pt home advantage and 30× win-rate-to-spread calibration are approximations that may systematically over- or under-price certain game types, inflating measured edge.

✗ No Series-State Encoding

In playoff games, whether a team is up 3–0 or 0–3 materially changes expected performance — via motivation, coaching strategy, and opponent pressure. The model treats Game 7 of a 3–3 series identically to a Game 1, which is factually wrong.

✗ Multicollinearity

Including both PM_DIFF and its components (HOME_PM_ROLL, AWAY_PM_ROLL) creates near-linear dependence. The L2 regulariser handles this imperfectly. A formal VIF analysis and feature selection step is absent.

✗ Flat Bet Sizing

All bets are $50 regardless of edge size. Kelly Criterion or a fractional Kelly rule would allocate more capital to higher-confidence bets — improving bankroll growth rate and Sharpe ratio in expectation.

✗ No Defensive Metrics

PLUS_MINUS is an aggregate box score stat that conflates offensive and defensive contributions. Including Defensive Rating, True Shooting%, and Assist/Turnover ratio would materially expand the information set available to the model.

Summary Assessment

This model demonstrates a working quantitative pipeline with correct temporal structure, principled hyperparameter selection, and multi-model comparison. The 66.54% accuracy is below the naive home-win baseline — a genuine limitation — but the probability calibration is sufficient to identify betting edges against a synthetic market, producing +42.91% ROI across 284 bets. The gap between "accurate classifier" and "calibrated probability estimator" is a known phenomenon in sports modelling: the market does not price games purely on expected win probability, so a model can lose on accuracy while winning on edge. The primary next steps — real odds data, series-state features, and Kelly sizing — would significantly sharpen the evaluation and potentially the edge.
